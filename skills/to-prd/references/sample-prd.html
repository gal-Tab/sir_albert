Event Lead Ingest Agent

Automate the ingestion of event contact CSVs from monday.com into HubSpot using the Submissions API. The agent normalizes inconsistent headers from three distinct source formats into high-fidelity HubSpot records, unlocking precise attribution and automated sales sequencing.

monday.com HubSpot API n8n 3 CSV formats Effort: M (3-5 days)

Definition of Done

All 3 CSV source formats are correctly detected, mapped, and ingested into HubSpot
monday.com item status updates to Done ✓ (no errors) or Done ⚠ [N] errors
Slack notification sent with final Success / Skip / Fail totals
Agent handles 100 rows in ~70 minutes with zero HubSpot rate limit violations

Out of Scope

✕ Deduplication of contacts already in HubSpot (handled by HubSpot's native dedup on email)
✕ Enrichment of contact data from third-party sources (Clearbit, ZoomInfo, etc.)
✕ Retroactive processing of historical event CSVs uploaded before agent launch

User Stories

1. As an event marketing manager, I want to upload a CSV of event contacts to a monday.com item, so that the contacts are automatically pushed into HubSpot without manual data entry.
2. As an event marketing manager, I want the agent to auto-detect which CSV format was uploaded (Standard, StrategyInsights, or Manual Scans), so that I don't have to manually select a template or rename columns.
3. As a sales rep, I want each contact's event lead notes and demo interest preserved in HubSpot, so that I have context when following up after the event.
4. As an ops manager, I want the monday.com item status to update automatically with a success/error count, so that I can see at a glance whether the ingest completed cleanly.
5. As an ops manager, I want a Slack notification with the final Success/Skip/Fail totals, so that I don't have to check monday.com to know the result.
6. As an ops manager, I want rows with invalid or missing emails to be skipped and logged (not crash the pipeline), so that one bad row doesn't block the rest of the batch.
7. As a sales rep, I want phone numbers normalized to E.164 format, so that HubSpot's calling features work out of the box.
8. As an admin, I want the agent to respect HubSpot's rate limits with a configurable delay, so that we never get throttled or banned.

▶ Normalization & Validation Rules

Email: Lowercase, trimmed. If missing or invalid (no @), row is skipped and logged to errors.

Phone (E.164): Prepend country-specific prefix (US/CA: +1, FR: +33, IL: +972) and strip all non-numeric characters except +.

Booleans: "Yes" maps to true; "No" or empty maps to false.

Notes: Maximum 65,000 characters. Truncate with ellipsis if exceeded.

▶ HubSpot API Configuration

Acceptance Criteria

1. GIVEN a Standard Template CSV is uploaded to a monday.com item, WHEN the agent runs, THEN all valid rows are submitted to HubSpot with correct field mapping.
2. GIVEN a CSV row has no email or an email without @, WHEN the agent processes it, THEN the row is skipped and the error is logged to the summary.
3. GIVEN a phone number "0541234567" with country "Israel", WHEN normalized, THEN the result is +972541234567.
4. GIVEN a Manual Scans CSV with a "Scanned by" column, WHEN the agent maps it, THEN the "Scanned by" column is skipped entirely.
5. GIVEN all rows are processed, WHEN the agent finishes, THEN the monday.com item status updates to "Done ✓" (0 errors) or "Done ⚠ [N] errors".
6. GIVEN the agent finishes processing, WHEN results are tallied, THEN a Slack message is sent with Success / Skip / Fail counts.
7. GIVEN 100 rows in a CSV, WHEN the agent runs with a 40-second delay, THEN processing completes in approximately 70 minutes with zero rate limit errors.
